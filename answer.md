# 智能招聘系统接入 AI Agent 平台（OpenClaw / Hermes）实施方案

## 一、背景与目标

当前项目已在 logic-grpc-service 中集成了 Eino ReAct Agent，HR 管理者可通过 HR 前端页面发起 AI 对话，查询岗位统计、筛选候选人等。未来方向是让所有系统都能与 AI Agent 交互和对话。本方案设计如何将该系统接入 OpenClaw 或 Hermes 等第三方 AI Agent 平台，使 HR 可通过即时通讯工具（飞书、钉钉、企业微信等）直接与招聘系统对话。

---

## 二、平台选型对比

### 2.1 OpenClaw 与 Hermes 对比

| 维度 | OpenClaw | Hermes Agent |
|------|----------|-------------|
| **定位** | 多消息平台网关 + Agent 编排框架 | 自进化 AI Agent，OpenAI 兼容 API |
| **核心优势** | 开箱即用的消息通道（飞书、钉钉、Slack 等） | 标准化 API，易于嵌入现有系统 |
| **部署方式** | Docker / npm，需自托管 | 本地 / 云部署，轻量级 |
| **工具注册** | Plugin/Skill 机制，Python/JS SDK | MCP 协议，或直接定义 tool schema |
| **与本系统集成方式** | 作为消息网关 + Agent 代理 | 作为 AI 后端 API，现有前端直接对接 |
| **适合场景** | 需要将系统能力暴露到 IM 工具（飞书/钉钉） | 需要标准化 AI API，复用现有前端 |

### 2.2 推荐策略

- **选 OpenClaw**：目标是通过飞书/钉钉等 IM 工具与招聘系统对话
- **选 Hermes**：目标是标准化 AI API 供多端调用
- **两者可串联**：OpenClaw 做消息接入，Hermes 做 Agent 推理

## 三、总体架构设计

### 3.2 通信链路说明

| 层级 | 组件 | 协议 | 职责 |
|------|------|------|------|
| 用户交互层 | HR 前端 / 飞书 / 钉钉 / 企微 | HTTP / WebSocket / IM 协议 | 用户发起对话和查看回复 |
| 消息接入层 | OpenClaw Gateway | Webhook / WebSocket | 统一 IM 通道接入、消息格式转换、路由分发 |
| AI Agent 层 | 招聘 Agent | LLM 推理 + 工具调用 | 意图识别、工具编排、自然语言回复组织 |
| 工具执行层 | MCP Server | MCP (SSE/stdio) | 标准化工具暴露、参数校验、结果返回 |
| 现有业务层 | web-gin / logic-grpc | gRPC / HTTP | 执行业务逻辑、数据持久化、权限校验 |

### 3.3 核心设计原则

1. **松耦合**：Agent 平台不与现有系统强绑定，通过标准化接口（MCP / HTTP）通信
2. **无侵入**：不改动现有业务代码，通过新增适配层（MCP Server）暴露能力
3. **权限隔离**：Agent 平台操作受现有 JWT/RBAC 权限约束，不绕过业务校验
4. **可观测**：所有 Agent 调用记录审计日志，与现有 `chat_history` 表打通

## 四、工具注册与暴露

### 4.1 工具分类总览

将招聘系统的业务能力抽象为以下工具，按操作类型分层：

| 分类 | 工具名称 | 功能描述 | 对应现有 API |
|------|---------|---------|-------------|
| **岗位管理** | `list_positions` | 查询岗位列表（支持分页/筛选） | GET /api/positions |
| | `get_position_detail` | 查看岗位详情 | GET /api/positions/:id |
| | `create_position` | 发布新岗位 | POST /api/hr/positions |
| | `update_position` | 编辑岗位信息 | PUT /api/hr/positions/:id |
| | `offline_position` | 下架岗位 | POST /api/hr/positions/:id/offline |
| | `online_position` | 上架岗位 | POST /api/hr/positions/:id/online |
| **候选人管理** | `list_candidates` | 查看某岗位的候选人列表 | GET /api/hr/positions/:id/candidates |
| | `get_candidate_detail` | 查看候选人详情 | GET /api/hr/candidates/:id |
| | `update_application_status` | 更新投递状态（通过/拒绝） | PUT /api/hr/applications/:id/status |
| **简历管理** | `download_resume` | 获取简历下载链接 | GET /api/hr/resumes/:id/download-url |
| **数据统计** | `get_application_stats` | 投递统计汇总 | 现有 Eino 工具 |
| | `get_position_stats` | 单岗位统计 | 现有 Eino 工具 |
| | `get_position_hotness` | 岗位热度排行 | 现有 Eino 工具 |
| **AI 对话** | `chat_with_ai` | 发起 AI 对话（保留现有 Eino Agent 能力） | POST /api/hr/ai/chat |

### 4.2 工具注册方式

**方案 A：HTTP 工具（推荐初始阶段使用）**

将现有 API 直接注册为 OpenClaw / Hermes 的 HTTP 工具。每个工具绑定到一个具体 API 端点：

```
工具定义：
- name: create_position
- description: "发布新的招聘岗位，HR 可以填写岗位名称、描述、要求、薪资范围和工作地点"
- input_schema:
    type: object
    required: [title, description]
    properties:
      title:        { type: string, description: "岗位名称，如 Go后端开发" }
      description:  { type: string, description: "岗位职责描述" }
      requirements: { type: string, description: "任职要求" }
      salary_min:   { type: number, description: "最低薪资(K)" }
      salary_max:   { type: number, description: "最高薪资(K)" }
      location:     { type: string, description: "工作地点" }
- endpoint: POST https://recruitment-api/internal/hr/positions
- auth: Bearer token (service account)
```

**优势**：无需新增服务，复用现有 web-gin-service 路由和权限校验
**劣势**：每次调用走完整 HTTP 链路，网络开销较大

**方案 B：MCP Server（推荐生产环境使用）**

基于 MCP（Model Context Protocol）协议实现专用工具服务器，作为 Agent 平台与招聘系统的桥梁。

```
┌─────────────┐     MCP (SSE/stdio)     ┌──────────────────────┐
│  OpenClaw / │◀──────────────────────▶│  recruitment-mcp-server │
│  Hermes     │                         │  (新增独立服务)        │
│  Agent      │                         │  端口 :8082           │
└─────────────┘                         └──────┬───────────────┘
                                               │ gRPC
                                               ▼
                                      ┌──────────────────┐
                                      │ logic-grpc-service│
                                      │   (:8081)         │
                                      └──────────────────┘
```

**优势**：
- 标准化协议，OpenClaw 和 Hermes 均原生支持 MCP
- 可复用现有 gRPC 调用，减少 HTTP 中间层
- 支持 streaming（适合 AI 对话场景）
- 工具发现机制（Agent 可动态发现可用工具）

**劣势**：需要新增一个 MCP Server 服务，增加部署复杂度

### 4.3 工具定义规范

每个工具定义必须包含以下要素：

| 要素 | 说明 | 示例 |
|------|------|------|
| **名称** | snake_case，一眼可理解 | `list_candidates` |
| **描述** | 详细说明功能和场景，影响 Agent 的决策 | "查询指定招聘岗位的所有候选人投递记录..." |
| **输入参数** | JSON Schema，标明必填/可选、类型、取值范围 | `{ position_id: number, status_filter?: string }` |
| **输出格式** | 统一结构 `{ success, data, error? }` | `{ success: true, data: { candidates: [...] } }` |

工具描述必须充分详细，因为 LLM Agent 完全依靠描述文字来决定何时调用哪个工具。例如：

```
工具 "list_candidates" 的描述应包含：
"查询指定招聘岗位的所有候选人投递记录，返回候选人姓名、学历、技能、投递状态等信息。
用于 HR 筛选候选人、查看投递进度。需要提供 position_id。
支持按状态筛选（pending/reviewed/accepted/rejected）。
返回结果包含 candidates 数组，每项含 candidate_id, real_name, education, skills, status。"
```

## 五、认证与授权设计

### 5.1 问题分析

现有系统的 API 均需 JWT 认证，JWT 中包含用户 ID 和角色（hr/candidate）。Agent 平台作为第三方调用时，必须解决"谁在操作"的问题——Agent 不能拥有超越其代理用户的权限。

### 5.2 认证方案：用户身份代理

核心思路：Agent 不代表自身操作，而是代理 HR 用户操作。用户通过 IM 工具先完成身份绑定，后续对话中 Agent 以该用户身份执行操作。

```
身份绑定流程：

  HR 用户                  IM 平台               OpenClaw Gateway            招聘系统
     │                       │                       │                       │
     │  ① 发送 /bind 招聘系统账号 │                       │                       │
     │──────────────────────▶│                       │                       │
     │                       │  ② 转发绑定请求        │                       │
     │                       │──────────────────────▶│                       │
     │                       │                       │  ③ 生成绑定验证码      │
     │                       │                       │────────────────────────▶│
     │                       │                       │  ④ 存储 user_id ↔ session │
     │                       │  ⑤ 返回验证码         │                       │
     │  ⑥ 收到验证码          │◀──────────────────────│                       │
     │◀──────────────────────│                       │                       │
     │                       │                       │                       │
     │  ⑦ 登录招聘系统网页，输入验证码进行绑定            │                       │
     │─────────────────────────────────────────────────────────────────────▶│
     │                       │                       │                       │
     │                       │                       │  ⑧ 绑定确认通知       │
     │                       │                       │◀────────────────────────│
     │                       │  ⑨ 绑定成功通知       │                       │
     │  ⑩ 绑定成功            │◀──────────────────────│                       │
     │◀──────────────────────│                       │                       │
```

### 5.3 授权流程（每次工具调用）

```
用户消息
    │
    ▼
OpenClaw Gateway ─── 识别 IM 用户身份
    │
    ▼
查询身份绑定关系 ─── 获取对应的招聘系统 user_id 和 JWT Token
    │
    ▼
调用工具（附带 JWT Token）
    │
    ▼
web-gin-service ─── 验证 JWT 身份 + role=hr
    │
    ▼
执行业务逻辑 ─── 权限校验、数据操作
    │
    ▼
返回结果给 Agent
```

### 5.4 双模式权限模型

建议创建两种调用模式以适应不同场景：

| 模式 | Token 类型 | 适用场景 | 权限范围 |
|------|-----------|---------|---------|
| **代理模式** | 用户绑定的 JWT | 具体用户操作（增删改） | 仅限于该用户拥有的资源 |
| **服务账号模式** | 独立 API Key | 只读查询、全局统计 | 仅读权限，不可变更数据 |

### 5.5 Token 管理

- **过期处理**：JWT 有过期时间（默认 7200s），工具调用前应检查 Token 有效性
- **自动续期**：在 MCP Server 层面维护 Token 刷新逻辑，当 Token 即将过期时使用刷新机制重新获取
- **安全存储**：用户绑定关系存储在招聘系统数据库中，OpenClaw 侧仅存储加密后的 session 标识

## 六、OpenClaw 具体接入方案

### 6.1 消息处理流程

```
  ① 用户（HR）在飞书/钉钉中向机器人发送消息
  ② 消息到达 OpenClaw Gateway
  ③ Gateway 根据 Binding 规则判定消息路由 → 转发给「招聘 Agent」
  ④ 招聘 Agent（LLM）解析用户意图 → 识别需要调用的工具
  ⑤ Agent 调用工具 → 通过 MCP Server 或 HTTP 请求招聘系统 API
  ⑥ 招聘系统返回结构化数据 → Agent 对数据做自然语言组织和润色
  ⑦ 回复流经 OpenClaw → 通过 IM 通道发送回用户 → HR 看到回答
```

### 6.2 关键配置项

| 配置项 | 说明 |
|-------|------|
| **Agent 定义** | 创建名为 "招聘助手" 的 Agent，配置 system prompt 描述招聘领域知识 |
| **Bindings 路由** | 将 HR 部门的飞书/钉钉用户或群聊绑定到该 Agent |
| **Tools/Skills 注册** | 注册 13 个招聘工具到 Agent，配置每个工具的 endpoint 和 schema |
| **LLM 模型配置** | 配置 DeepSeek（或 Claude/千问）作为 Agent 推理模型 |
| **Memory 记忆** | 开启会话记忆，保持多轮对话上下文连贯性 |
| **会话超时** | 设定空闲超时（如 30 分钟），超时自动清理上下文 |

### 6.3 System Prompt 设计

Agent 的 system prompt 是决定对话质量的关键，应包含以下内容：

1. **身份定义**：你是智能招聘系统的 AI 助手，帮助 HR 管理者高效完成招聘全流程工作
2. **能力边界**：
   - 可以：查询岗位/候选人/投递信息、统计数据、创建/编辑岗位、更新投递状态
   - 不可以：修改候选人资料、删除岗位数据、直接查看简历内容、修改系统配置
3. **工具使用规则**：
   - 查询统计类优先调用数据统计工具
   - 涉及多个步骤的复杂查询（如"Go 岗位的候选人都有谁"）需先查岗位 ID 再查候选人
   - 工具调用失败时应如实告知用户，不编造数据
4. **隐私声明**：
   - 候选人简历属于敏感个人信息，Agent 仅返回文件名和格式，不读取文件内容
   - 联系方式等个人隐私仅在用户明确询问并提供合理理由时返回
5. **多轮上下文**：
   - 当用户使用"他们""那个岗位""之前说的"等代词时，从前文中寻找引用
   - 已确认的岗位名称、候选人信息应当在后续对话中保持

### 6.4 IM 通道集成

OpenClaw 天然支持以下通道，需为每个通道注册机器人应用：

| 通道 | 注册方式 | 关键注意事项 |
|------|---------|-------------|
| **飞书** | 飞书开放平台创建自建应用 → 获取 AppID/AppSecret → 配置 Webhook/WebSocket | 需开通机器人权限、消息推送权限，配置事件订阅 URL |
| **钉钉** | 钉钉开放平台创建机器人 → 获取 ClientID/ClientSecret → 配置回调 URL | 需配置 robotCode，开通 stream 模式或 HTTP 模式 |
| **企业微信** | 企业微信创建应用 → 获取 CorpID/AgentID/Secret → 配置回调 URL | 需配置可信域名，消息加解密需配置 EncodingAESKey |

### 6.5 会话与状态管理

- **Session 映射**：OpenClaw 的每个会话映射到招聘系统中的一个 `session_id`，格式复用现有 `fmt.Sprintf("%d", time.Now().UnixNano())`
- **对话记录同步**：OpenClaw 中的聊天记录可选择通过回调接口写入招聘系统的 `chat_history` 表，保持数据一致性
- **上下文恢复**：用户重新进入对话或长时间断连后，加载最近的对话历史作为上下文预热

## 七、Hermes Agent 具体接入方案

### 7.1 消息处理流程

```
  ① 外部系统（HR 前端 / OpenClaw）调用 Hermes API
  ② Hermes Agent 接收请求 → 维护会话上下文
  ③ Agent（LLM）解析用户意图 → 调用已注册的工具
  ④ 工具通过 MCP 或 HTTP 调用招聘系统
  ⑤ 招聘系统返回结构化数据 → Agent 组织自然语言回复
  ⑥ 通过 SSE 流式返回或完整 JSON 响应返回
```

### 7.2 接入方式对比

| 接入方式 | 说明 | 适用场景 |
|---------|------|---------|
| **直接 API 调用** | 现有前端直接调用 Hermes `/v1/chat/completions` | 替代现有 Eino AI 对话（保留前端交互） |
| **MCP 工具注册** | 招聘系统通过 MCP Server 暴露工具，Hermes 动态加载 | 保留现有前端，增强 Agent 能力 |
| **双轨并行** | Hermes 调用 Eino Agent 作为子工具（tool calling） | 保留 Eino Agent 的 ReAct 逻辑不变 |

### 7.3 推荐集成路线

以 MCP 方式为例：

1. 部署并启动 Hermes Agent，启用 API Server（默认端口 `:8642`）
2. 在 Hermes 配置中启用 MCP 客户端，连接到 recruitment-mcp-server（`:8082`）
3. 配置代理认证，通过 `service_token` 字段注册 JWT 获取方式
4. 现有前端或 OpenClaw 通过 OpenAI 兼容格式调用 Hermes：
   ```
   POST http://hermes:8642/v1/chat/completions
   Authorization: Bearer <api_server_key>
   {
     "model": "hermes-agent",
     "messages": [
       {"role": "system", "content": "你是招聘系统 AI 助手..."},
       {"role": "user", "content": "Go 岗位有多少候选人？"}
     ],
     "stream": true
   }
   ```

### 7.4 与现有 Eino Agent 的关系

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| 保留现有 Eino 对话能力 | Eino Agent 作为子工具被 Hermes 调用 | 不改动现有逻辑，复用已有的 4 个 SQL 工具 |
| 替换为更强的 Agent | Hermes 直接对接 MCP Server | 统一技术栈，减少维护成本 |
| 共存过渡 | 两者并行，按功能路由 | 风险最低，逐步迁移 |

## 八、数据隐私与安全

### 8.1 敏感数据分级管控

| 数据类型 | 风险等级 | 处理措施 |
|---------|---------|---------|
| 候选人简历（姓名、电话、学历、经历） | **高** | 仅在工具调用时传递摘要字段，不将完整简历内容送入 LLM 上下文 |
| 岗位需求描述 | **中** | 可送入 LLM 但需标注公司内部信息，提醒 Agent 注意 |
| 投递状态、统计数据 | **低** | 可正常使用 |
| 用户密码 | **极高** | 任何时候不传递给 Agent 平台，不存在任何工具中 |

### 8.2 安全措施清单

1. **工具调用审计**：所有 Agent 发起的工具调用记录到审计日志，记录内容包括：
   - 谁调用的（IM 账号 + 系统 user_id）
   - 何时调用的（时间戳）
   - 调用了什么工具（工具名称）
   - 传入了什么参数（参数快照）
   - 返回了什么数据（结果摘要）
   - 调用耗时

2. **数据最小化原则**：工具返回值只包含满足查询需求的最小字段集，不返回多余数据。例如查询候选人列表时，如果不涉及下载简历，就不返回 `resume_id`。

3. **简历内容隔离**：Agent 不直接读取简历二进制内容，仅获取元数据（文件名、格式、大小）。简历下载必须通过签名 URL 流程，且链接有时效性（3600s）。

4. **传输安全**：
   - IM 通道使用 HTTPS/WSS 加密
   - MCP Server 与 Agent 之间配置 TLS
   - 招聘系统 API 仅对内网暴露，不开放公网直接访问

5. **操作二次确认机制**：
   ```
   用户："把张三的投递状态改为拒绝"
   Agent："确认要将候选人 张三（岗位：Go后端开发）的投递状态改为 已拒绝 吗？"
   用户："确认"
   Agent → 调用 update_application_status 工具 → 返回结果
   ```

### 8.3 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| LLM 幻觉导致误操作 | 调用了错误的工具或传入了错误参数 | 工具输入 schema 严格校验 + 敏感操作二次确认 |
| JWT 过期 | 工具调用失败 | 在 MCP Server 层面实现 Token 自动续期 |
| 简历敏感数据泄露 | 候选人隐私泄露 | Agent 硬限制不读取简历内容，仅传元数据 |
| IM 平台 API 限流 | 消息发送延迟 | 配置指数退避重试策略 |
| Agent 上下文过长 | Token 消耗高、回复质量下降 | 自动摘要 + 滑动窗口 + 上下文压缩 |
| 多轮上下文指向混淆 | 错误关联岗位/候选人 | 每次工具调用返回关键 ID，Agent 显式确认 |

## 九、对话流程与状态管理

### 9.1 典型对话场景推演

**场景一：统计查询**

```
用户："帮我看看这个月 Go 岗位的投递情况"

Agent 推理链路：
  ① 用户想查询 Go 相关岗位的投递数据
  ② 调用 list_positions(filters: {title: "Go"})
     → 返回 [{id: 1, title: "Go后端开发"}, {id: 2, title: "Go高级工程师"}]
  ③ 对每个 position_id 调用 get_position_stats
     → position 1: {total: 20, reviewed: 5, rejected: 2, accepted: 0}
     → position 2: {total: 15, reviewed: 3, rejected: 1, accepted: 0}
  ④ 汇总数据，组织自然语言回复

Agent 回复：
"Go 相关岗位共有 2 个，总投递 35 份：
  1. Go后端开发 — 20 份投递（已面试 5 人）
  2. Go高级工程师 — 15 份投递（已面试 3 人）
  需要我进一步查看某个岗位的候选人详情吗？"
```

**场景二：候选人筛选**

```
用户："Go后端的候选人有哪些是硕士学历的？"

Agent 推理链路：
  ① 用户指定了 Go后端开发 岗位，从前文可知 position_id=1
  ② 调用 list_candidates(position_id: 1)
     → 返回候选人数组（含 education 字段）
  ③ 过滤出 education="硕士" 的候选人
  ④ 格式化结果

Agent 回复：
"Go后端开发 候选人中硕士学历的有 5 位：
  1. 王小明 - 北京大学 - 计算机科学 - 技能：Go, K8s, Redis
  2. 李小红 - 武汉大学 - 软件工程 - 技能：Go, MySQL, Docker
  ...
  需要进一步联系或查看简历吗？"
```

**场景三：业务操作（含二次确认）**

```
用户："下架 Go高级工程师 岗位"

Agent 推理链路：
  ① 用户要求下架岗位，属于敏感操作
  ② 先调用 list_positions(filters: {title: "Go高级工程师"})
     → 确定 position_id
  ③ 不直接执行，而是发起二次确认

Agent 回复：
"确认要将「Go高级工程师」岗位下架吗？下架后候选人将无法搜索和投递该岗位。"
用户："是的，下架吧"
Agent → 调用 offline_position(position_id: 2) → 返回结果
Agent 回复："「Go高级工程师」已成功下架。你可以随时通过上架操作重新发布。"
```

### 9.2 会话生命周期管理

```
新消息到达
    │
    ▼
┌──────────────────┐
│  会话查找/创建    │ ← 判断是否已有活跃 session，没有则创建
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│  上下文加载       │ ← 从 Memory 加载历史，从 chat_history 恢复上下文
└──────┬───────────┘
       │
       ▼
┌──────────────────────────────┐
│  Agent 推理与工具调用循环      │
│  ┌─────┐  ┌─────┐  ┌─────┐  │
│  │思考  │─▶│工具  │─▶│观察  │─▶...→ 最终回复
│  └─────┘  └─────┘  └─────┘  │
└──────────┬───────────────────┘
           │
           ▼
┌──────────────────┐
│  回复用户 + 持久化  │ ← 回复写入 IM 通道，同时保存到 chat_history
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│  等待下一轮消息    │ ← 设定 TTL（空闲超时后清理上下文）
└──────────────────┘
```

### 9.3 上下文管理策略

| 策略 | 说明 | 实现方式 |
|------|------|---------|
| **滑动窗口** | 保留最近 N 轮对话（如 20 轮） | Agent 平台原生支持 |
| **自动摘要** | Token 接近上限时压缩历史 | 调用 LLM 对历史对话做摘要总结 |
| **关键信息持久化** | 将岗位名称、候选人等关键实体提取到 Memory | 使用 Agent 平台的 Memory 功能 |
| **对话历史回填** | 每次对话后异步写入 `chat_history` 表 | 通过回调或消息队列批量写入 |

### 10.3 监控指标与告警

| 监控项 | 指标 | 告警阈值 | 说明 |
|-------|------|---------|------|
| Agent 调用量 | 工具调用次数/分钟 | > 100/min | 检测异常流量激增 |
| 工具调用成功率 | 成功数 / 总调用数 | < 95% | 检测系统故障或参数错误 |
| 调用响应时间 | p99 延迟 | > 5s | 检测性能瓶颈 |
| IM 消息到达率 | 投递成功数 / 总发送数 | < 99% | 检测 IM 通道异常 |
| Token 消耗 | 每次对话 Token 数 | > 10K/次 | 检测上下文是否异常增长 |
| JWT 过期率 | 过期导致的失败 / 总调用 | > 5% | 检测 Token 续期机制故障 |

### 10.4 异常处理流程

```
工具调用失败
    │
    ├─ 网络超时 ───→ 重试 3 次（指数退避）→ 仍失败 → 告知用户"系统繁忙，请稍后再试"
    │
    ├─ 认证过期 ──→ 自动刷新 Token → 重试 → 刷新失败 → 要求用户重新绑定
    │
    ├─ 参数错误 ──→ 检查输入 → 修正后重试 → 仍错误 → 要求用户提供更准确的信息
    │
    └─ 业务错误 ──→ 将错误信息返回 Agent → Agent 用自然语言解释给用户
```

## 十二、关键设计决策总结

| 决策点 | 选择方案 | 决策理由 |
|--------|---------|---------|
| 工具暴露方式 | **MCP Server（方案 B）** | 标准化协议、Agent 平台无关、支持流式、支持工具动态发现 |
| 认证模式 | **用户身份绑定 + 服务账号双模式** | 既支持个人化增删改操作，又支持全局只读查询 |
| 消息通道 | **OpenClaw Gateway** | 开箱支持飞书/钉钉/企微，社区活跃，维护成本低 |
| Agent 推理模型 | **DeepSeek（复用现有）或 Claude** | DeepSeek 成本低、已对接；Claude 推理质量更高 |
| 对话持久化 | **双写（IM 平台 + chat_history 表）** | 保持数据一致性，支持跨平台对话恢复 |
| 简历内容访问 | **Agent 不读取二进制内容** | 隐私安全红线，不可妥协 |
| 部署方式 | **Docker Compose 统一编排** | 现有服务已适配容器化，新增组件无缝接入 |
| 实施策略 | **分 4 阶段渐进式交付** | 基础查询先上线验证，再逐步开放写操作和 IM 通道 |

## 十三、总结

本方案设计了智能招聘系统接入 OpenClaw / Hermes 等三方 AI Agent 平台的完整架构和实施路径。核心思路是通过新增 MCP Server 作为标准化工具代理层，将现有 26 个 API 抽象为 13 个 AI 可调用的工具，OpenClaw 负责多消息通道接入和 Agent 编排，最终实现 HR 通过飞书/钉钉等即时通讯工具以自然语言与招聘系统交互的目标。方案遵循松耦合、无侵入、权限隔离和可观测四大设计原则，覆盖了工具注册、认证授权、数据安全、对话管理、部署运维等全链路细节。
